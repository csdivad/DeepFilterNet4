# Investigation: `pipeline_awesome` Acoustic Performance

## The Issue
The `pipeline_awesome` pathway is not producing noticeably improved speech or eliminating background noise in the output audio files generated by `scripts/compare_models.py`, despite the intent to use Voice Activity Detection (VAD) to pinpoint speech and suppress all other noise.

## Root Cause Analysis

The core issue stems from an architectural disconnect between how VAD is utilized during **training** versus how the model operates during **inference**.

### 1. VAD is a Training Loss Weight, Not an Inference Gate
In `DeepFilterNet/df_mlx/training_losses.py`, the `_compute_pipeline_awesome_losses` function calculates a VAD proxy (`p_ref`) based on the energy of the *clean* audio.

```python
# DeepFilterNet/df_mlx/training_losses.py (Lines 800-810)
# VAD proxy is calculated from clean audio
p_ref = _compute_vad_proxy(clean_mag, ...)

# It is then used to weight the speech loss
proxy_frame_3d = mx.expand_dims(p_ref, axis=1)
speech_loss = mx.mean(mx.abs(out_log - clean_log) * mask * proxy_frame_3d)
```

This means VAD is used strictly as a **loss multiplier** during training. It tells the model: *"Pay more attention to minimizing the error when speech is present."*

### 2. Inference Lacks Explicit VAD Gating
During inference (which `compare_models.py` uses via `model.enhance()`), the model relies entirely on its learned weights to generate a mask.

```python
# DeepFilterNet/df_mlx/model.py (Lines 1286-1350)
def enhance(self, audio: mx.array) -> mx.array:
    # ... STFT ...
    spec, _, _ = self(spec) # Forward pass (Encoder -> Backbone -> Decoder -> Mask)
    # ... iSTFT ...
```

There is **no explicit VAD module** running during inference to detect speech and force the output to zero in non-speech regions. The model is expected to have *implicitly* learned to act as a VAD and suppress noise, but without an explicit architectural gate, it often fails to achieve the hard suppression desired.

### 3. Loss Formulation May Not Penalize Noise Enough
The `pipeline_awesome` loss heavily weights the `speech_loss` using the VAD proxy, but it may not be applying a strong enough penalty to the *noise* regions. If the loss function doesn't aggressively penalize the model for outputting non-zero values when `p_ref` is low (no speech), the model won't learn to aggressively suppress background noise.

## Proposals for Investigation and Confirmation

To confirm these findings and understand the model's behavior better, we can perform the following investigations:

1. **Visualize the Learned Masks vs. VAD Proxy:**
   - Modify a test script to output the raw masks generated by the model during inference.
   - Compare these masks against a ground-truth VAD mask (e.g., using Silero VAD or the energy-based proxy).
   - *Hypothesis:* The model's masks will likely remain relatively high (close to 1) even in non-speech regions, indicating it hasn't learned the VAD behavior implicitly.

2. **Analyze Loss Component Contributions:**
   - During a short training run, log the individual contributions of the `speech_loss` (weighted by VAD) versus the noise/background loss components.
   - *Hypothesis:* The loss penalty for failing to suppress noise in non-speech regions is too small compared to the penalty for distorting speech.

## Proposed Fixes

Once confirmed, here are specific fixes we can implement, ranging from training tweaks to architectural changes:

### Fix 1: Modify the Training Loss (Implicit Learning)
Adjust `_compute_pipeline_awesome_losses` to explicitly and heavily penalize non-zero outputs in non-speech regions.
- **How:** Add a specific "silence loss" term: `silence_loss = mx.mean(mx.abs(out_log) * (1.0 - proxy_frame_3d)) * heavy_weight`.
- **Pros:** No changes needed to the inference architecture; keeps the model fast.
- **Cons:** The model still has to learn this behavior implicitly, which can be difficult depending on capacity.

### Fix 2: Add an Explicit VAD Gate During Inference (Architectural)
Integrate a lightweight VAD (like the energy proxy or a small dedicated network) directly into the `enhance` method in `model.py`.
- **How:** Calculate `vad_mask` during `enhance()` and multiply the final output spectrogram by this mask before the iSTFT.
- **Pros:** Guarantees hard suppression of background noise in non-speech regions.
- **Cons:** Adds computational overhead during inference; requires tuning the VAD threshold for inference.

### Fix 3: Multi-Task Learning (Architectural + Training)
Add a VAD classification head to the model's backbone.
- **How:** Train the model to explicitly predict the VAD probability alongside the enhancement mask. During inference, use this predicted VAD probability to gate the enhancement mask.
- **Pros:** Forces the model's internal representations to learn speech detection explicitly, likely improving both the mask quality and providing a built-in gate.
- **Cons:** Requires modifying the model architecture (`model.py`) and the training loop to handle the new classification head and loss.
